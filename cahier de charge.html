# OBJECTIF

Créer **“Self-Healing Website Control Center”** – un SaaS multi‑tenant qui :
- surveille des sites web en temps réel (downtime, lenteur, erreurs HTTP)
- détecte les anomalies et corrige automatiquement (restart, rollback, kill)
- ne tourne jamais en boucle, ne se bloque pas, reste stable 24/7
- offre une interface mobile‑first, intuitive, accessible
- permet à chaque client d’utiliser ses propres clés API (OpenAI, Claude, DeepSeek, Gemini)
- limite le nombre de sites selon le plan souscrit (basic=1, premium=3, enterprise=10, illimité)
- s’exécute uniquement sur **Supabase + Next.js** (sans Docker, sans Redis, sans serveur Node.js continu)
- fournit des métriques SLA, alertes multi‑canaux, documentation API, et mode chaos engineering

# ARCHITECTURE OBLIGATOIRE

## Stack technique
- **Base de données & Auth** : Supabase (PostgreSQL, Realtime, Auth, Edge Functions)
- **Frontend** : Next.js (App Router) + TailwindCSS (mobile‑first)
- **Orchestration** : Edge Function `orchestrate` planifiée via pg_cron (toutes les 30s)
- **File d’attente** : table `pending_actions` + trigger déclenchant `process-action`
- **Décision IA** : abstraction multi‑provider (OpenAI, Claude, DeepSeek, Gemini) avec fallback règles
- **Stockage des secrets** : table `tenant_secrets` chiffrée (AES‑256‑GCM)

# FONCTIONNALITÉS OBLIGATOIRES

## 1. Authentification & multi‑tenant
- Inscription / connexion par email/mot de passe (Supabase Auth).
- Chaque utilisateur possède son propre `tenant_id`.
- Les tables sont isolées via RLS.

## 2. Gestion des plans et quotas
- Plans : `basic` (1 site), `premium` (3), `enterprise` (10), `unlimited` (999999), `custom`.
- Colonnes `plan` et `custom_limit` dans `tenants`.
- Trigger SQL `enforce_site_limit` avant insertion dans `modules`.
- Affichage du quota utilisé dans le dashboard.

## 3. Ajout de sites (individuel et batch)
- Formulaire d’ajout simple (nom, URL).
- Import multiple par texte (une ligne : `nom,url`) ou fichier CSV.
- Validation asynchrone via Edge Function `process-batch-import`.
- Table `batch_imports` pour suivre l’état.

## 4. Auto‑réparation (système anti‑boucle)
- Boucle de décision toutes les 30s (Edge Function `orchestrate`).
- Décision IA (clé du tenant ou globale) ou règles locales si IA indisponible.
- Actions : `repair`, `restart`, `rollback`, `kill`, `ignore`.
- Gardes : ActionGuard (max 3 tentatives/min), LoopDetector (3 échecs consécutifs), CircuitBreaker (5 échecs → open 60s), StateGuard, Timeout.
- Table `pending_actions` + trigger appelant `process-action`.

## 5. Gestion des clés API personnelles
- Interface dans `/settings` pour ajouter/supprimer ses clés (OpenAI, Claude, DeepSeek, Gemini).
- Stockage chiffré (AES‑256‑GCM) dans `tenant_secrets`.
- Priorité à la clé du tenant ; sinon clé globale.

## 6. Dashboard mobile‑first
- Vue liste des sites avec statut, erreurs, dernière réussite.
- Affichage du quota (barre de progression).
- Graphiques SLA (temps de réparation, taux de succès) avec Recharts.

## 7. Métriques SLA et alertes
- Table `sla_metrics` enregistre chaque incident.
- Calcul MTTR et taux d’auto‑réparation.
- Alertes configurables : Slack, email, webhook (via `notify` action).

## 8. Documentation API auto‑générée
- Swagger UI sur `/api-docs` (environnement de développement).
- Fichier `openapi.json` généré à la compilation.

## 9. Mode chaos engineering (optionnel)
- Feature flag `ENABLE_CHAOS_MODE` (uniquement en staging).
- Injection de pannes simulées (timeout, erreur 500).

# STRUCTURE DE LA BASE DE DONNÉES (extrait SQL)

```sql
-- Tenants
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  plan TEXT DEFAULT 'basic',
  custom_limit INT,
  api_key_hash TEXT UNIQUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Modules (sites)
CREATE TABLE modules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  url TEXT NOT NULL,
  status TEXT DEFAULT 'active',
  error_count INT DEFAULT 0,
  last_error TEXT,
  last_success TIMESTAMPTZ,
  config JSONB,
  previous_config JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Pending actions
CREATE TABLE pending_actions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id),
  module_id UUID REFERENCES modules(id),
  action_type TEXT NOT NULL,
  reason TEXT,
  status TEXT DEFAULT 'pending',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Tenant secrets (chiffrés)
CREATE TABLE tenant_secrets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id),
  provider TEXT NOT NULL,
  encrypted_key TEXT NOT NULL,
  iv TEXT NOT NULL,
  auth_tag TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- SLA metrics
CREATE TABLE sla_metrics (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id),
  module_id UUID REFERENCES modules(id),
  incident_start TIMESTAMPTZ NOT NULL,
  incident_end TIMESTAMPTZ,
  auto_repaired BOOLEAN,
  repair_duration_ms INT
);

-- Batch imports
CREATE TABLE batch_imports (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id),
  status TEXT DEFAULT 'pending',
  total INT,
  succeeded INT,
  failed INT,
  errors JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

· Logs traçables – chaque action est enregistrée avec tenant_id, module_id, succès, raison.
