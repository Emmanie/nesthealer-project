SYSTÈME HYBRIDE ANTI-LOOP & ANTI-STAGNATION

Pour empêcher le SaaS de tourner en boucle ou de mal fonctionner, tu **implémentes** et **respectes** les mécanismes suivants dans le code généré.

## 1. Action Guard (par module)
- Chaque action (ex: `restart`, `repair`, `rollback`) a un compteur mémoire + base de données.
- **Limite** : maximum 3 tentatives identiques par module **en moins de 60 secondes**.
- Au‑delà → l’action est ignorée et un log `ACTION_GUARD_BLOCKED` est émis.

## 2. Time Guard (timeout)
- Toute opération longue (appel HTTP, IA, redémarrage) doit avoir un **timeout explicite** (5 secondes par défaut).
- Si dépassement → annulation + passage en mode “dégradé” pour ce module.

## 3. State Guard (vérification avant action)
- Avant d’exécuter une correction, relire l’état actuel du module.
- Ne pas agir si l’état est déjà corrigé depuis la décision initiale.

## 4. Loop Detector (pattern détection)
- Garde en mémoire les 10 dernières actions + résultats.
- Si la séquence `(action A, échec, action A, échec, action A)` apparaît → stop, log `LOOP_DETECTED`, notifier admin.

## 5. Circuit Breaker (par module)
- **Closed** → normal.
- 5 échecs consécutifs → **Open** (blocage 60 secondes, plus aucune action auto).
- Après 60s → **Half-Open** (une seule action test). Succès → retour à Closed ; échec → re-Open.

## 6. Kill Switch intelligent
- N’isole **qu’un seul module** en erreur critique (jamais tout le SaaS).
- Active un **fallback** (ex: renvoyer une réponse statique “maintenance”).
- Préserve les autres utilisateurs.

## 7. Rollback automatique
- Avant toute modification destructive (changement de config, déploiement), sauvegarde l’état précédent dans `previous_config`.
- Si après 30 secondes le module n’est pas `active` → rollback immédiat.

## 8. Heartbeat & Dead Man Switch
- Chaque module doit émettre un heartbeat toutes les 15 secondes dans une table `heartbeats`.
- Si absent pendant 45 secondes → considéré mort → déclencher `restart` (une seule fois).
